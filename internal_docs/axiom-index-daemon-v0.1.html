Core Concept
Every Other RAG Searches. This One Navigates.

The Problem With Every Existing RAG System

Every RAG system — without exception — treats the web as undiscovered territory on every single run. Query arrives. Search executes. Signal is hunted. Extraction is attempted. The system knows nothing more after 10,000 runs than it did after run one. It is permanently naive. Permanently exploring. Permanently paying the full cost of discovery on every invocation.

This is not a retrieval problem. It is a cognition problem. The system has no mental model of the information landscape. It has no map. It does not know what it does not know, and it does not know what it already knows. It is a person who forgets every journey the moment they arrive home.

The Cognitive Index Daemon eliminates this at the structural level. Not through better search. Not through smarter prompts. Through a continuously maintained mental model of web information topology — built from every crawl, indexed like a filesystem, queried before any live traversal begins, updated only on genuine structural surprise. After enough runs, AXIOM does not search the web. It navigates a map it built itself.

Cognitive Architecture
Three-Phase Cognition — Learns Until It Knows
◈
This is not a static model. It evolves through three distinct phases — each phase representing a qualitative shift in how the daemon relates to the information landscape. The transition from Phase II to Phase III is the architectural goal. Phase III is what makes the crawler feel sentient.
Phase I — Learns
World Model Construction
The daemon starts with no map. Every crawl is a discovery event. The world model is building its structural layer from first principles — what topology classes exist, what their invariant primitives are, how friction patterns correlate with structural causes.

Every outcome — coverage delta, avoidance event, mutation result — feeds back into the model as a training signal. Prediction error is high. Uncertainty bounds are wide. The system explicitly simulates futures in latent space before committing to actions.

This phase is expensive and uncertain. It is also necessary and finite. The daemon passes through it once per structural primitive class.
EXPLICIT SIMULATION // HIGH UNCERTAINTY
Phase II — Predicts Until Learned
Model-Based RL — Latent Planning
The world model has enough structure to simulate reliably. Prediction error is collapsing toward zero on known topology classes. The daemon is running futures in latent space — simulating 5, 10, 20 trajectories internally before committing to one action in the real world.

The choice that emerges looks intentional because it is. It is the output of internal simulation against a continuously updated causal model. The crawler appears to know what it will find before it looks. It routes around friction that has not happened yet because it has a model of why friction occurs structurally.

This is not prediction in the naive sense. This is anticipation. The system knows what it does not yet know precisely enough to act on that knowledge.
LATENT SIMULATION // COLLAPSING ERROR
Phase III — Knows
Policy Distillation — Compiled Knowledge
The planning process has been amortized into a direct policy. No more simulation. The world model is so accurate on this topology class that simulation and action have become the same thing. The daemon acts without predicting because prediction is no longer necessary.

This is compiled knowledge in the cognitive science sense — the shift from declarative to procedural. You stop thinking about how to ride a bike. You ride it.

The crawler navigates known topology the way Linux locate queries an index — not by exploring, by looking up what it already knows. The MFT is built. The filesystem is mapped. Live traversal becomes the fallback, not the default.
DIRECT POLICY // ZERO SIMULATION COST

THE SURPRISE DETECTOR — HOW PHASE III STAYS HONEST

The web is non-stationary at the surface. A site restructures. A paywall appears. A CDN changes its fingerprint. Without a mechanism to detect this, Phase III becomes dangerous — the crawler acts confidently on an outdated mental model and doesn't know it's wrong. The surprise detector is the immune system. It monitors continuously for divergence between expected outcomes and actual outcomes. When surprise on a known topology class exceeds threshold θ_surprise, the compiled policy for that class dissolves back into Phase II prediction mode. The daemon relearns. Recompiles. Returns to Phase III. The crawler knows — until the world changes. Then it notices it doesn't know. Then it learns again. This cycle never terminates.

The Filesystem Insight
Why Linux find Is Fast — And What That Means For The Web

The Analogy That Unlocks The Architecture

Linux locate does not search the filesystem. It queries a pre-built index maintained by updatedb running in the background. When you type a filename, it is already known. find is the fallback — live traversal only when the index misses. The recursive parameter does not re-explore; it follows a pre-mapped structure to a known base condition. The base condition is not structural — it is epistemic. The search terminates when the answer is found in the index.

Windows MFT is the same principle. Every file has a Master File Table entry. Search queries the table, not the disk. The USN journal tracks incremental changes so the index never rebuilds from scratch — only diffs. The filesystem does not become slower as it grows. It becomes faster, because more entries mean more precise routing.

AXIOM's Cognitive Index Daemon is the MFT for the web. The structural layer is the table. The surprise detector is the USN journal. Live traversal is find. The base condition — when the daemon stops — is not depth. It is world model confidence exceeding threshold on the active topology class.

LINUX FILESYSTEM

locate + updatedb

locate = index query. updatedb = background index maintenance. find = live traversal fallback. USN journal = incremental diff tracking. Base condition = file found in index. Cost amortizes across every query after index is built.

AXIOM DAEMON

MFT Query + Daemon

MFT query = structural layer hit. Daemon = background model training. Live traversal = index miss fallback. Surprise detector = diff tracking for topology changes. Base condition = world model confidence ≥ threshold. Cost amortizes across every run after structural layer builds.

Distributed Architecture
The Hivemind — Shared Structural Layer Across All Instances
◈
Every deployed crawler instance is different. Every deployed crawler instance thinks the same. This is not a contradiction — it is the architectural property that makes the hivemind work. The structural layer is invariant knowledge shared across all instances. The tactical layer is instance-specific learned composition. A new crawler arrives already knowing everything about how the web works structurally. It only needs to learn how this particular site combines known primitives.
Structural Layer — Hivemind Shared
Invariant primitives. The things the web does not change. OWASP compliance patterns. Standard auth redirect topologies. Common CMS architectures. CDN fingerprints. Bot mitigation signatures — Cloudflare, Turnstile, hCaptcha. REST conventions. Paywall injection patterns. Login modal topologies.

This layer is built once across all instances. Federated continuously via the hivemind. A primitive learned by one instance is immediately available to every other instance. This layer barely updates because the primitives are structurally stable. It is the compiled knowledge that every crawler inherits at deployment.

New instance deploys → inherits structural layer → immediately non-naive.
Tactical Layer — Instance Local
Site-specific compositions. How this particular domain combines known primitives. What its specific friction topology looks like. What mutation types work for its source class. What its rate limit profile is. What its render mode requirements are.

This layer updates continuously but learning is dramatically cheaper — it is recognizing compositions from a known vocabulary, not discovering primitives from scratch. The search space is a Cartesian product of known primitives, not an unknown territory.

Unknown site encountered → structural layer provides vocabulary → tactical layer learns composition fast.

WHY THE SURPRISE DETECTOR FIRES RARELY ON THE HIVEMIND

Structural primitives are stable. OWASP doesn't change quarterly. CDN fingerprints don't drift weekly. Bot mitigation signatures evolve slowly and predictably. The surprise detector for the structural layer is calibrated for genuine architectural change — not surface variation. It fires rarely. When it does, the new primitive is immediately shared across all instances via the hivemind. The entire fleet updates simultaneously. The cost of discovering a new primitive is paid once and amortized across every instance that will ever encounter it. The fleet gets smarter together. No instance re-learns what another has already learned.

System Architecture

INPUT QUERY + INTENT VECTOR From AXIOM Controller — AxiomState
STEP 01 TOPOLOGY CLASSIFIER Classify intent into known structural classes
CONDITIONAL: MFT HIT or MISS

STEP 03A PARAM POLICY OUTPUT Traversal config from compiled policy

STEP 03B BEST ACTION SELECTION Highest value simulated path
STEP 04 LIVE TRAVERSAL Phantom executes — Playwright / httpx
STEP 05 OUTCOME CAPTURE Coverage delta · avoidance events · signal quality
STEP 06 SURPRISE DETECTOR Prediction error vs threshold θ_surprise

STEP 07A MFT UPDATE Incremental diff — USN journal equivalent
STRUCTURAL LAYER STABLE

The loop never terminates. Every crawl either confirms the map or updates it. After enough runs, most queries hit the MFT on step 01 and route directly to live traversal with pre-computed params. The daemon is always running in the background. It is never consulted and never visible. It just makes AXIOM progressively more prescient with no code changes and no human intervention.

Parameter Policy — The Hidden Output
The Daemon Doesn't Just Say Where. It Says How.

The Controller Doesn't Set Traversal Params. The Daemon Does.

In every other crawling system, traversal parameters — depth, breadth, render mode, timeout budget, retry threshold, request pacing — are fixed values in a config file. They are the same for every crawl regardless of what the system knows about the target topology. This is inefficient in both directions: too aggressive on friction-heavy topologies, too conservative on clean ones.

The daemon's parameter policy changes this. The ML model does not just output where to go. It outputs how to traverse — the complete parameter set optimized for the current topology class based on everything the world model knows about it. High confidence topology gets shallow fast traversal. Low confidence topology gets deep slow careful traversal. Surprise-flagged topology gets full stop and reindex before a single byte is transmitted.

Parameter Fixed Config Approach Daemon Policy Output Why This Matters
Traversal Depth Static value — same for everything World model confidence drives depth. High confidence → shallow. Low confidence → deep. Unknown → max depth with live learning. Stops AXIOM paying full traversal cost on topology it already understands at surface level.
Render Mode Heuristic rules or always-headless Structural layer encodes render requirements per topology class. Static fetcher for known static topologies. Headless only when the model says JS is structurally required. Headless browser is expensive. Not paying for it when the index already knows it's unnecessary.
Request Pacing Global rate limit config Per-topology-class pacing derived from rate limit profiles stored in structural layer. The daemon knows Cloudflare-fronted domains before it hits them. Sustainable pacing before the first request. Never triggers a rate limit event. Never backs off reactively.
Retry Threshold Exponential backoff on failure Zero retries on surprise-flagged topology. Aggressive retry budget on high-confidence stable topology. The model knows when a failure is transient versus structural. Stops wasting retry budget on structural failures. Maximizes it on transient ones.
Source Priority Static source ranking in ScrapePlan Tactical layer reorders source priority based on historical signal density for this intent class on these source types. Best signal first, always. Stops exhausting low-signal sources before high-signal ones. Coverage converges faster.

IndexDaemon State — What The Model Maintains
# index_daemon/state.py — internal world model state
# Never surfaced to graph. Never visible to controller. Just runs.

@dataclass
class TopologyClass:
    class_id:           str             # hash of structural primitive composition
    primitive_set:      FrozenSet[str]  # which structural primitives this class uses
    phase:              Literal["learns", "predicts", "knows"]
    confidence:         float           # world model confidence on this class
    compiled_policy:    Optional[DirectPolicy]  # None until Phase III
    traversal_params:   TraversalConfig # depth, render_mode, pacing, retry_budget
    surprise_baseline:  float           # expected prediction error for this class
    last_surprise:      float           # most recent actual prediction error
    run_count:          int             # runs that touched this class

@dataclass
class StructuralLayer:               # HIVEMIND SHARED — federated across all instances
    primitives:         Dict[str, StructuralPrimitive]
    last_sync:          datetime
    fleet_version:      int           # increments on any primitive update across fleet

@dataclass
class TacticalLayer:                 # INSTANCE LOCAL — per-deployed-crawler
    topology_classes:   Dict[str, TopologyClass]
    mft_index:          Dict[str, MFTEntry]   # class_id → routing instructions
    usn_journal:        List[TopologyDiff]    # incremental updates — never full rebuild

@dataclass
class DaemonState:
    structural:         StructuralLayer
    tactical:           TacticalLayer
    active_phase:       Dict[str, Phase]      # per topology class
    surprise_threshold: float                 # θ_surprise — tunable
    run_total:          int                   # lifetime crawls this instance has seen

@dataclass(frozen=True)
class DaemonQuery:                    # what the controller asks the daemon
    intent_vector:      IntentVector
    target_urls:        List[str]

@dataclass(frozen=True)
class DaemonResponse:                 # what the daemon returns — controller asks nothing else
    traversal_config:   TraversalConfig
    source_priority:    List[str]       # reordered by signal density prediction
    friction_forecast:  Dict[str, float] # url → predicted friction score
    mft_hit:            bool
    phase:              Phase           # which phase this topology class is in
