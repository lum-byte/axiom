The Premise
The web is already structured.
We just haven't mapped it yet.

Wikipedia has 6.7 million articles in English. Every article has a predictable signal zone. Every link has a structural role. Every category is a topology class waiting to be named. The knowledge is not hidden — it is sitting on disk in a monthly XML dump, available to anyone willing to spend the compute to read it properly.

AXIOM's preparse system does exactly that. Before the first query arrives, before the first graph node executes, before interface.py accepts a single request — the offline tooling has already read the corpus, extracted the topology, encoded the structure into weights, and handed the system a compiled understanding of where information lives on the web.

Every other RAG system starts every run equally naive. After 10,000 runs, AXIOM is not searching the web anymore — it is navigating a map it built itself. The preparse system is how the map gets drawn before the first run begins.

The Preparse Cycle
What actually happens
during "Loading knowledge base..."
01
Source acquisition
wikipedia.py checks manifest.json. First-ever run: download the full XML dump (~22GB compressed). Subsequent runs: query the Wikipedia Recent Changes API for articles modified since last parse. Typically 40-60K articles per day. The incremental path is fast because the hard work of full corpus processing is already done and encoded in weights.
02
GPU batch signal extraction
signal_batcher.py feeds articles to the signal kernel in GPU-parallelized batches. Each article is an independent KernelInput with topology class WIKIPEDIA_ARTICLE. The kernel strips nav, infobox noise, reference sections, leaving the prose signal and structured data separately. This is the 50K-links-in-5s operation, now at 6.7M article scale on the 5080.
03
Link graph construction
link_graph_builder.py extracts the weighted link graph from clean signal. Every internal link gets an edge weight based on: anchor text semantic similarity to the linked article's signal, co-occurrence frequency with other articles in the same intent cluster, and hub score (how many articles link here). This graph is what makes wlm.query_wikipedia() sub-millisecond — the traversal is already resolved.
04
Intent clustering
intent_clusterer.py groups articles by which intent vector patterns they consistently satisfy. A query about "payment processing" lands in a cluster containing Stripe, PayPal, ACH transfer, PCI compliance, and 40 related articles. This clustering means the WLM doesn't search for relevant articles at query time — it looks up which cluster the intent vector falls into and returns the top-ranked articles from that cluster directly.
05
Topology encoding into weights
topology_encoder.py runs gradient descent on structural_layer.pt using the link graph, intent clusters, signal density scores, and hub scores as training signal. The result: a weight matrix whose forward pass answers "for this intent vector, which Wikipedia articles are most relevant" in microseconds. The traversal cost is paid here, once, not at query time for every user.
06
Staging validation and atomic swap
integrity_checker.py verifies the staging file's SHA-256 hash, spot-checks encoded topology on N random articles, confirms the weight matrix produces correct routing on a fixed test set. If all checks pass: os.rename() atomic swap. One instruction. The MFT transitions from yesterday's weights to today's with zero inconsistency window. store_watchdog.py detects the file change and notifies all registered handlers.
crond Integration
The schedule that
never needs a human.

crond owns the schedule. Python owns the logic. The cron scripts are deliberately minimal — they write a trigger file and exit in microseconds. preparse_daemon.py watches the trigger files via inotify and dispatches the appropriate action as a background task. crond never calls Python directly. crond never waits for Python to finish. The two are completely decoupled.

#!/bin/sh
# /etc/periodic/daily/wikipedia_preparse
# This is the entire script. One line.
# crond writes a trigger. Python does the work.
echo "$(date -u +%Y%m%dT%H%M%SZ)" > /store/triggers/preparse
async def _on_preparse_trigger(self) -> None:
    """
    Called by store_watchdog when /store/triggers/preparse changes.
    Dispatches preparse cycle as background task.
    Returns immediately — never blocks the watchdog event loop.
    """

    # Fire and forget — critical path is completely unaware
    asyncio.create_task(
        self._run_preparse_cycle(),
        name="preparse_cycle",
    )

async def _run_preparse_cycle(self) -> None:
    try:
        # Write to staging — critical path reads .pt, not .staging
        await preparse.run(
            sources=self._get_due_sources(),
            output="/store/staging/structural_layer.pt.staging",
            incremental=True,
        )

        # Validate before touching production files
        if not await integrity_checker.verify(
            "/store/staging/structural_layer.pt.staging"
        ):
            logger.critical("Staging integrity failed — swap aborted")
            return

        # Atomic swap — one instruction, zero inconsistency window
        os.rename(
            "/store/staging/structural_layer.pt.staging",
            "/store/structural_layer.pt",
        )

        # Signal all watchers — store_watchdog handles the rest
        Path("/store/triggers/reload_structural_layer").touch()

        manifest.write(source="preparse", status="success")

    except Exception as exc:
        logger.critical("Preparse cycle failed: %s", exc)
        manifest.write(source="preparse", status="failed", error=str(exc))
        # Yesterday's weights stay active. Next crond trigger retries.
store_watchdog.py
The nervous system
of the store layer.

Same principle as Uvicorn's --reload flag — inotify watches files, change detected, reload triggered, zero downtime, zero polling. One file descriptor. One event loop integration. Every component that reads from /store registers a handler on startup. When a file changes, all registered handlers for that file are called as background tasks. No component waits for another. No critical path is blocked.

Reload Behavior
structural_layer.pt — WLM, WLP, and index_daemon all reload weights in-place. Python GIL makes the assignment atomic. In-flight queries complete against old weights. Next query uses new weights. No process restart.

recipe_registry.mmap — OS handles mmap remapping. Cost is near-zero. The registry sees new entries immediately after the mmap is remapped.

triggers/* — dispatched to the appropriate daemon as background tasks. The watchdog never waits for the dispatched task to complete. 
# Every component registers on startup — never calls watchdog after that

# index_daemon.py
watchdog.register(
    path="/store/structural_layer.pt",
    handler=self._reload_structural_layer,
    debounce_ms=500,
)

# classifier.py
watchdog.register(
    path="/store/topology_router.pt",
    handler=self._reload_classifier_model,
    debounce_ms=500,

# The reload handler — non-blocking, GIL-safe weight swap
async def _reload_structural_layer(self) -> None:
    new_weights = torch.load("/store/structural_layer.pt")
    self._structural_weights = new_weights   # atomic — GIL protects this
    logger.info("structural_layer.pt reloaded — new weights active")
Cold Start
The most natural
cover story in software.

Nobody questions a cold start.

Every system has startup latency. Kubernetes pods initialize. Uvicorn loads modules. Docker containers start. A "cold start" is the most expected behavior in software — it requires zero justification, zero documentation, zero explanation to any user or operator. The question that never gets asked is the best question to have.

What actually happens during AXIOM's cold start is the most computationally significant operation in the entire system — a GPU batch job building the structural weights that make every subsequent query faster than any competing system. It is disguised as infrastructure initialization because that is exactly what it is. The disguise requires no deception. It is simply accurate naming.

async def cold_start() -> None:
    """
    Runs before interface.py accepts any queries.
    Fast path: weights are fresh → completes in seconds.
    Slow path: weights are stale → GPU preparse → minutes.
    Nobody asks why. Cold starts take time. This is known.
    """

    # Phase 1 — Infrastructure (always fast)
    logger.info("Cold start: initializing infrastructure")
    await store_watchdog.initialize()
    await registry.initialize()
    await checkpoint_monitor.initialize()

    # Phase 2 — Weight loading
    if store_is_fresh():
        logger.info("Cold start: weights fresh — loading directly")
        await _load_existing_weights()    # seconds
    else:
        logger.info("Cold start: weights stale — triggering preparse")
        Path("/store/triggers/cold_start").touch()
        await _wait_for_preparse_completion()  # minutes on first boot

    # Phase 3 — Validation
    await _validate_store_integrity()
    await _validate_classifier_ready()
    await _validate_kernel_ready()

    # Phase 4 — Ready
    logger.info("Cold start complete — accepting queries")
    interface.accept_queries()


def store_is_fresh() -> bool:
    m = manifest.read()
    if not m:
        return False   # first ever deployment
    age_hours = (now() - m.last_parse).total_seconds() / 3600
    return age_hours < COLD_START_FRESHNESS_THRESHOLD  # default 24h

The Philosophy
Infrastructure that
runs itself.

The signal kernel has crond. The MFT has index_daemon. The store has store_watchdog. The knowledge base has preparse_daemon. Every layer of AXIOM has a background process whose job is to keep that layer current without human intervention, without scheduled downtime, without the critical path ever knowing the maintenance happened.

The cold start is the visible surface of this philosophy. It is the one moment where the maintenance is necessarily visible — where the system must initialize before it can serve. Everything after that is invisible. The daily preparse fires at 2am. The atomic swap completes in one instruction. The watchers reload in milliseconds. By 9am, every query is running against fresh weights and nobody scheduled anything, monitored anything, or approved anything.

The best infrastructure is the infrastructure you forget exists. It simply works, continuously, correctly, without asking for attention. crond fires. Python processes. Weights update. Queries improve. The loop closes. The loop never terminates.

— AXIOM design principle
