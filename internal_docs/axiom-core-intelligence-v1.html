01 — Core Premise
TAG not RAG — The Fundamental Inversion
The Problem with RAG

Retrieval-Augmented Generation assumes the hard problem is retrieval and embedding. It throws vector databases, chunking strategies, and semantic search at a black box web it has never learned to read. The cleaning is an afterthought: strip some HTML tags, maybe run a readability heuristic, hope for the best. Every query starts from zero. The system never gets smarter.

RAG believes meaning is proximity in vector space. A filing cabinet metaphor dressed in mathematics. Store everything. Retrieve by similarity. The web is an undifferentiated mass of bytes to be ingested, chunked, and embedded without understanding what any of it structurally is.

The TAG Inversion

Topology-Addressed Generation assumes the hard problem is knowing what the page is before you touch it. The WLM classifies topology before bytes arrive. When the extraction pipeline runs, it already knows: Is this a REST JSON endpoint or a Docusaurus doc page? Is there a paywall wrapper polluting the DOM? Is the code block lazy-loaded or server-rendered? What depth of link traversal is actually signal vs noise?

TAG believes meaning is location in structure space. A navigation metaphor. The web is a typed system with known structural classes. Inference infrastructure predicts those classes before the fetch. The extraction is topology-class-aware — not generic stripping, not statistical heuristics, but a compiled policy executing against known structure.

RAG — What it does
Retrieves by cosine similarity. Meaning = proximity in vector space. Correct by accident, not by structure.
Every query starts fresh. Query 1 and query 1,000,000 are equally expensive. System never compounds.
Generic HTML cleaning. BeautifulSoup, trafilatura — 50-200ms, still retains nav bars, cookie banners, sidebar cruft. Doesn't know what it's looking at.
Index grows forever. More content = larger index = slower retrieval. Cost scales with knowledge.
Hits the same Cloudflare wall 100,000 times. No memory. No routing intelligence. No phase learning. Same failure mode forever.
Chunking for embeddings. Retrieval quality limited by chunk quality. No retrieval step can fix bad cleaning.
TAG — What AXIOM does
Routes by structural address. Meaning = location in topology space. Correct by architecture, not by chance.
Every query teaches the system. Phase I → II → III earned through use. Query 1,000,000 is faster and cheaper than query 1.
Topology-class-aware extraction. Shell-native grep/sed/awk pipeline. 7.4ms. 99.27% compression. Knows exactly what it's looking at before it looks.

02 — Philosophical Foundation
Two People, One Web
The Foundational Analogy

When you meet a person — do you wonder anxiously what kind of person they are, what you should know about them, what to say, what they might ask, what would be nice to take so it's easier to learn from them next time you meet? Or do you hard-judge on first contact, extract what you can, and run the same extraction script the next 100,000 times — falling into every honeypot, getting rejected by every locked door you already knew was locked, never learning that the same hole keeps catching you?

The second person is every scraper ever built. They hit the same Cloudflare wall 100,000 times and learn nothing. They get rate limited on stripe.com at 9:00am and try again at 9:00:01am. They ignore robots.txt because they weren't told not to. They fall into the same honeypot link on every crawl because it looked like content.

AXIOM is the first person. Before approaching, it has already thought: what kind of site is this? what does it want from me? what's the right way to ask? what's worth taking and what's noise they want me to carry so I slow down? The WLM's friction forecast is that social read happening in under 2ms. And the structural layer is memory — AXIOM has met stripe.com before, knows how it thinks, knows where the real content lives versus the marketing wrapper.

The deep part: "what would be nice to take so it's easier to learn the next time" — that's the hidden state updating through training. Every crawl that surprises the model fires a SurpriseEvent, dissolves the cached assumption, and rebuilds it smarter. Traditional RAG doesn't get better at reading stripe.com on visit 100,000 than visit 1. AXIOM does.

The Linux Kernel Analogy

TAG is literally a kernel for web intelligence. The Linux kernel manages hardware, abstracts complexity, provides a syscall interface, schedules processes — applications think in files, not disk sectors. AXIOM manages web structure, abstracts topology, provides a topology class interface, routes queries — callers think in intent vectors, not HTML bytes.

An application on Linux makes a syscall. The kernel handles everything below that. It does not ask the application to understand interrupt handlers, memory-mapped IO, or filesystem journaling. The application asks for a file; it gets content.

The AXIOM graph sends a DaemonRequest with an intent vector. TAG handles everything below that — classification, traversal policy, friction forecasting, fetch mode selection, recipe compilation, extraction. The graph asks for signal; it gets clean structured content. The kernel does not exist at the graph level.

03 — Three-Phase Cognition
Phases Earned, Not Assigned

Every topology class starts at Phase I. Phases are earned through accumulated confidence, not configured by operators. The system self-promotes based on what it has learned. A topology class at Phase III means AXIOM has accumulated enough structural knowledge about that class that it can route without live simulation — compiled policy, direct execution, zero exploration cost.

The critical invariant: Phase III can regress. A SurpriseEvent that fires dissolve on a Phase III class sends it back to Phase II — the world changed, the compiled policy is stale, revalidation required. Phases track the current truth about what the system knows, not what it once knew.

05 — The MFT
Weights as Index — The Core Insight
topology_router.pt is not a model that uses an index. It IS the index.
Forward pass IS the lookup. Writing to index IS the gradient step. Same file viewed from two directions.
At Query Time
Query arrives → intent_vector → one matrix multiply against hidden_state (1, 256) → output is TraversalPolicy + SourceHints

O(1) lookup. Not search — memory read. No database. No cosine similarity. No approximate nearest neighbors.
At Training Time
Domain topology event → tokenize → Mamba SSM forward pass → gradient step via index_daemon → hidden_state updated after 500ms consistency window

Writing to index = gradient step. Every crawl makes the routing smarter.
As a File
Binary PyTorch state dict. ~50-80MB. Fixed size forever. After 6.7M Wikipedia articles, after 50K crawl cycles, the hidden state is still (1, 256). 256 floats encoding everything the system knows about where answers live on the web.

# The MFT lookup — O(1), one matrix multiply:
def readout(self, intent_vector=None) -> ReadoutResult:
    with torch.no_grad():  # enforced: hidden_state cannot change here
        h = self.hidden_state  # (1, 256) — the entire MFT
        if intent_vector is not None:
            h = h + self.intent_projection(intent_vector)
        topology_logits  = self.topology_head(h)   # (1, 18)
        traversal_raw    = self.traversal_head(h)  # (1, 7)
        friction_raw     = self.friction_head(h)   # (1, 5)
        source_embedding = self.source_head(h)     # (1, 512)
    return ReadoutResult(topology_logits, traversal_raw, friction_raw, source_embedding)
What topology_router.pt IS
  • The complete WLM weights including persistent hidden_state buffer
  • The MFT — (1, 256) tensor encoding accumulated structural knowledge of the web
  • A routing oracle: forward pass = routing lookup
  • A learning artifact: gradient step = index update
  • Fixed-size forever regardless of crawl history length
  • Written ONLY by index_daemon.py via staging + atomic rename
  • Read with weights_only=True — no exceptions, anywhere
What topology_router.pt is NOT
  • Not a vector database — no cosine similarity, no approximate search
  • Not a cache — no expiry, no eviction, no TTL
  • Not a knowledge base storing facts about the web
  • Not a configuration file — no topology class definitions or URL patterns
  • Not append-only — overwritten entirely on each checkpoint via atomic rename
  • Not human-readable — binary format, belongs in .gitignore
  • Not shared directly across fleet — each instance has its own MFT
06 — World Latent Model
Satellite View — Mamba SSM
What the WLM Is

The WLM is a Mamba State Space Model maintaining a compressed latent representation of the structural topology of the web. It is the satellite view — it sees domains, topology classes, friction patterns, structural relationships from above. It answers one question before every fetch: Given what I know about how the web is structured, how should AXIOM approach this specific topology class at this specific moment in time?

The WLM has three simultaneously active identities: during inference it is a routing oracle (one matrix multiply, O(1), faster than any database lookup); during training it is a sequence model (Mamba SSM processing domain topology events, updating its world model); as a file it is topology_router.pt — a checkpoint of accumulated structural knowledge that survives restarts and picks up exactly where it left off.

Why Mamba SSM and Not Transformer or GNN

Transformer: Self-attention is O(n²) in sequence length. Sequence length = millions of domain events = not viable. No persistent state — each forward pass sees only its context window. Cannot accumulate structural knowledge across millions of crawl cycles.

GNN: Treats topology classes as static graph nodes — topology classes evolve. Requires full graph at inference time — WLM needs incremental updates. No temporal ordering — the sequence of domain events matters.

Mamba SSM: O(n) training, O(1) inference. Fixed-size hidden state regardless of history length — (1, 256) whether 1 or 10 billion events have been seen. Selective state transitions — Mamba learns which domain events matter. Perfect semantic alignment with MFT concept: the MFT IS a compressed structural map, the SSM hidden state IS a compressed structural map — the same thing by definition.

# The problem: hidden_state update during in-flight readout = inconsistent forward pass
# The solution: exact counter with 500ms ceiling (not asyncio.sleep(0.5))

async def update_hidden_state(self, new_hidden: torch.Tensor) -> None:
    deadline = time.monotonic() + 0.500  # 500ms ceiling
    while self._model._active_readout_count > 0:
        if time.monotonic() > deadline:
            log.warning("hidden_state_update_deadline_exceeded")
            break
        await asyncio.sleep(0.001)  # 1ms poll

    # Atomic reference swap — Python GIL makes this safe
    self._model._router.hidden_state.copy_(new_hidden)
    self._model._hidden_state_version += 1

# If readouts complete in 3ms: update happens in 3ms. Not 500ms.
# 500ms is only reached if something is wrong with an in-flight query.
# asyncio.sleep(0.5) would waste 497ms every time. This wastes 0.
07 — World Latent Parser
Street Level — The Cartographer
WLP is Not a Classifier — It is a Cartographer

Every other system asks "what is this page about?" WLP asks "what is the shape of this page?" — and then draws the map. Signal zones, noise zones, boundaries, structural topology. Not semantics. Cartography.

The analogy: Google Maps didn't become accurate because someone drew every street. It became accurate because millions of drivers drove those streets and the aggregate signal corrected the map. WLP does the same for web structure. Every crawl cycle is a driver. Every CleanSignalEvent is a GPS ping. The ZoneMap is the map that emerges from aggregate structural signal.

The MFT in hidden weights part is what makes it genuinely novel. Standard cartography stores the map in a database. WLP stores it in weights. The map IS the model. Reading the map IS the forward pass. Updating the map IS the gradient step. Same artifact, viewed from two directions.

08 — Signal Kernel
Alpine Strip — The Extraction Engine
What the Signal Kernel Is

The signal kernel is a two-process Alpine Linux container running a grep/sed/awk pipeline against topology-class-specific recipes. It is the only component that touches raw page bytes. It knows nothing about the web, nothing about topology classes, nothing about the WLM. It receives raw content and a recipe. It runs the recipe. It returns clean signal. That is all.

The Alpine container has network_mode: none. It cannot make network calls. It cannot phone home. It is isolated by design. The warm container bug is fixed: the container stays alive between invocations (Process 1), with the grep pipeline and crond checkpoint daemon running as Process 2. Startup latency: ~120ms cold, ~0ms warm.

11 — Graph Algorithms
Emerged from First Principles — Not Chosen
The Remarkable Thing

AXIOM did not implement these algorithms deliberately. It arrived at them from first principles because the underlying problem — efficient routing over a large graph with hierarchical structure — is the same problem these algorithms solve. When a system is architecturally correct, the right algorithms appear inside it as consequences, not decisions.

Contraction Hierarchies
CH — Shortest Path via Node Importance Ordering
In CH, less important graph nodes are "contracted" — shortcuts are added between their neighbors so queries skip them entirely. The hierarchy exists for fast shortest-path computation.

AXIOM's implementation:
PARENT_CLASS_MAP is the contraction hierarchy. FORUM_THREAD → BLOG_POST → NEWS_ARTICLE. When no recipe exists for FORUM_THREAD, the system doesn't search the whole graph — it walks the contracted hierarchy to the nearest hub with a known policy. registry.py implements recursive chain walk with cycle guard, never single-step.
Hub Labelling
HL — O(1) Distance Queries via Precomputed Hub Sets
HL precomputes hub labels for high-centrality nodes. Distance between any two nodes = shortest path through a shared hub. Query time: O(1) hub set intersection.

AXIOM's implementation:
Phase III topology classes ARE hubs. Pre-compiled policies, maximum confidence, direct routing. cold_start_warmup() precomputes hub labels (L2 cache entries) for all 18 known topology classes before queries begin. Phase III query hits a hub — one cache lookup, O(1).
ALT Algorithm
A* + Landmarks + Triangle Inequality
ALT uses preselected "landmark" nodes to compute admissible heuristics for A* search. The triangle inequality on landmark distances gives a lower bound: dist(u,v) ≥ |dist(u,L) - dist(v,L)|.

AXIOM's implementation:
Surprise events create landmarks. When a topology class fires dissolve, it becomes a reference point — the traversal policy avoids or routes around it. The triangle inequality gives a lower bound on structural distance between unknown topology and known landmarks. FrictionForecast is the admissible heuristic.
Multilevel Decomposition
MLD — Hierarchical Graph Partitioning for Routing
MLD partitions the graph into hierarchical cells. Queries within a cell use local routing; queries across cells use precomputed cell-level shortcuts. The hierarchy is the optimization.

AXIOM's implementation:
The phase system IS multilevel decomposition. Phase I = full graph search (exploring the whole cell). Phase II = partition-aware search (routing uses cell-level shortcuts from WLM predictions). Phase III = single cell lookup (the partition is the compiled policy — no graph traversal needed).
12 — Security Architecture
MFT Guardian + Fetch Containment
The Threat Model

Weight injection is the ML equivalent of SQL injection. An attacker compromises the gradient pipeline and writes hostile tensors directly into topology_router.pt. The MFT absorbs it. AXIOM routes wrong at O(1) speed and nobody knows why — no error, no signal, just a system that has learned to route toward attacker-controlled content.

The architectural fix: separation of write authority. A small privileged MFT guardian process is the only thing on the system with write authority to topology_router.pt. Index_daemon never touches the file directly. It sends proposed updates through a narrow typed channel. The guardian runs RLCPC validation independently before writing anything. Full compromise of index_daemon gets you the ability to read the MFT. You cannot write to it.

Alpine Signal Kernel
network_mode: none — unchanged
The base. Rigid. Untouchable. Zero network access — not restricted, genuinely none. No process can add network access to this container. Receives raw content via stdin. Returns clean signal via stdout. The kernel does not know the internet exists.
gVisor Fetch Container
User-space Linux kernel — nested isolation
Nested Linux inside the fetch layer — gVisor (runsc) or Firecracker microVM. Every syscall from internet-facing code passes through gVisor's user-space kernel, never the host kernel. Allowed syscalls: read, write, connect, send, recv, close. Allowed outbound: 80/443 only. Allowed filesystem writes: /tmp/fetch_staging/ only. Allowed processes: fetcher binary only.

Output to TAG layer: typed HMAC-signed envelope via Unix domain socket only. No raw bytes. No arbitrary data. One message type. Anything else: rejected at the gate.
Breach Detector + Honeypot Flip
Real-time behavioral monitoring
Breach signals: unexpected syscall, outbound on port other than 80/443, write outside /tmp/fetch_staging/, fork/exec of unexpected binary, inbound connection attempt.

On breach detected:
  1. Snapshot current container state
  2. Spin up honeypot from snapshot — attacker sees identical environment
  3. Flip iptables: redirect traffic from breached container → honeypot
  4. Kill real fetch container, spin up clean replacement
  5. Log everything the attacker does in honeypot
Attacker thinks they're still in your system. They're in a jar. Real system is already clean.
MFT Guardian
mft_guardian.py — ~200 LOC, sole write authority
Separate container, separate user, no network, store volume mounted read-write only to guardian, read-only to everything else including index_daemon. Guardian accepts one operation through a narrow channel: a typed, HMAC-signed, RLCPC-blessed state dict diff. Not a file path. Not arbitrary bytes. One typed message. Everything else: rejected.

Guardian independently runs RLCPC validation before writing. If validation fails: rejected, logged, no write. Full index_daemon compromise = read access to MFT only. Write path does not exist from that process.
13 — RLCPC
Reinforced Learning through Controller Policy Check
The Invisible Layer

RLCPC is not a gate you walk up to and present credentials. It is not an API you can call. It is not a permission system you can query. It has no public interface. It sits between every signal and every output and makes decisions silently. Nobody knows it's there. Nobody can probe it. Nobody can ask it what its rules are.

contracts.py validates existence — if a thing was built, it was built correctly. RLCPC validates meaning — if a thing wants to touch the MFT, it must prove it deserves to. contracts.py is the dead monolith: static, written once, never changes at runtime. RLCPC is the living monolith: same authority, but it breathes. Every gradient step, every phase transition, every routing decision — nothing commits until RLCPC blesses it.

15 — Non-Negotiable Rules
Architectural Invariants
The Laws — Enforced by Architecture, Not Convention
  1. Classifier never fetches. Classification happens before the request. URL + headers + 4KB window is all it ever sees.
  2. Parser never executes. The recipe compiler produces shell recipes. It never runs them. Execution is the kernel's job.
  3. Sanitizer never uses LLM. Last-mile strip is deterministic. No model call, no inference, no non-determinism.
  4. Surprise fires on divergence, not failure. A failed fetch is not surprising. A page that looks nothing like the WLM's prediction is surprising. These are different signals.
  5. index_daemon async fire-and-forget. The RL loop never blocks the query path. Training happens in the background. The loop never terminates.
  6. Phases earned, not assigned. No operator configures a topology class to Phase III. Confidence thresholds enforce this. Phase skipping (1→3) is architecturally prohibited.
  7. interface.py written absolutely last. The only public surface. Written after every component it exposes is complete, tested, and verified. Not a day earlier.
  8. Crawler bus has zero logic. Emit. Subscribe. That is all. Business logic in the bus is a design error.
  9. All store writes use staging + atomic rename. No partial writes. No file that is half-valid. The rename is the commit.
  10. WLM + WLP always asyncio.gather(), never sequential. They are peers. One waiting on the other is a design error. Always parallel.
  11. hmac.compare_digest — never ==. Timing-safe comparison on every HMAC check. Without exception.
  12. os.fsync on dead letter writes. A dead-lettered event must survive a process crash. The fsync is the durability guarantee.
  13. readout() never modifies hidden_state. Not sometimes, not usually — never. torch.no_grad() + no assignment. This is enforced by architecture, not by discipline.
  14. forward() only via WLMTrainingInterface. The sentinel prevents direct construction. index_daemon uses only the training handle. No inference path has access to forward().
  15. weights_only=True on all torch.load() calls. No exceptions. Security requirement. Not a performance hint.

18 — Intent-Conditioned Extraction
What No RAG System Has Ever Done
The Problem With Every System Before This

Every RAG system, every web scraper, every search engine encodes user intent as a query string. Flat text. Loses all structure. The retrieval step finds semantically similar chunks but has no idea what part of those chunks matters for this specific intent. The LLM receives the whole chunk — including the parts that are noise for what the user actually needs — and has to filter it at generation time.

AXIOM does something categorically different. Intent is encoded as a 256-dimensional float32 vector that travels through the entire pipeline as a first-class citizen. Every component that touches the query sees the full structured intent — not a string, not keywords, not a URL-encoded parameter. The intent shapes the extraction before a single byte reaches the LLM. The LLM never has to filter noise because the recipe already filtered it.

This is not a retrieval improvement. It is a different problem being solved — precision extraction at the fetch layer, not semantic filtering at the generation layer.

How Every Other System Works
Query string encoding. "twilio authy lost password recovery codes" → flat string → semantic search → retrieve page → hand user the whole page
Retrieval finds the page. User still has to find the answer on the page. The system's job ends at retrieval.
LLM filters noise. Chunk contains API reference, getting started guide, pricing info, and the recovery codes section. LLM has to ignore 90% of what it received.
Cannot exclude. "NOT api_reference NOT pricing" is a hack search engines partially honor. There is no structured exclusion.
Same extraction every time. Same page retrieved for "lost password" and "API rate limits" produces the same chunk. Intent blind.
How AXIOM Works
Vector encoding. intent_vector[256] with primary, secondary, exclude, urgency, user_state dimensions → travels through every component as structured data
Extraction finds the answer. The recipe compiles to extract exactly what the intent needs. The system's job ends at delivering the answer, not the page.
Recipe filters noise. intent.exclude suppresses zones at recipe compilation time. Excluded zones don't exist in the output — not filtered by LLM, never generated.
Structured exclusion. intent.exclude = ["api_reference", "pricing", "getting_started"] actively suppresses those ZoneMap zones. First-class pipeline concept.
Intent-conditioned recipe per query. Same page, different intent vector → different recipe → different extraction output. Same topology class, precision-targeted output.
